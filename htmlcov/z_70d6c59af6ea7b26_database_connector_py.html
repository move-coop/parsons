7class DatabaseConnector(ABC): 

8 """ 

9 An abstract base class that provides a uniform interface for all Parsons database connectors. 

10 This class should be used in functions instead of the specific database connector classes 

11 when the functions don't rely on database-specific functionality. 

12 

13 It ensures that any class that inherits from it implements the methods that are uniform 

14 operations when working with databases. 

15 

16 Should you use `DatabaseConnector` instead of `Redshift`/`BigQuery`/etc? 

17 

18 Overall this class is mostly useful for code in the Parsons library, not code using it. 

19 There could be some exceptions. In general though, if you are writing a script to do a task 

20 like moving data out of an API service and into a data warehouse, you probably do not need 

21 to use DatabaseConnector. You can probably just use the Parsons class that directly corresponds 

22 with the database that you use. 

23 

24 Here are more examples of situations where you may or may not need to use DatabaseConnector: 

25 

26 1. You do not use type annotations, or you don't know what "type annotations" are - No 

27 

28 If you do not use type annotations for your code, then you do not need to think about 

29 `DatabaseConnector` when writing your code. This is the most common case. If none 

30 of the cases below apply to you, then you probably don't need it. 

31 

32 In this simple example, we are not using type annotations in our code. We don't need 

33 to think about exactly what class is being passed in. Python will figure it out. 

34 

35 ```python 

36 def my_database_function(db): 

37 some_data = get_some_data() 

38 db.copy("some_table", some_data) 

39 

40 # These will all just work: 

41 my_database_function(Redshift()) 

42 my_database_function(MySQL()) 

43 my_database_functon(BigQuery()) 

44 ``` 

45 

46 2. You only use one database in your work - No 

47 

48 This is where most people will fall. Usually code is not intended to run on 

49 multiple databases without modification. For example, if you are working for 

50 an organization that uses Amazon Redshift as your data warehouse, you do not 

51 need to use `DatabaseConnector` to write ETL scripts to load data into your 

52 Redshift. It is rare that organizations switch databases. In the cases where 

53 that does occur, usually more work is required to migrate your environment and 

54 your vendor-specific SQL than would be saved by using `DatabaseConnector`. 

55 

56 3. You are writing a sample script or a tutorial - Yes 

57 

58 If you are using Parsons to write a sample script or tutorial, you should use 

59 `DatabaseConnector`! If you use `DatabaseConnector` type annotations and the 

60 `discover_database` function, then your sample code will run on any system. 

61 This makes it much easier for new programmers to get your code working on 

62 their system. 

63 

64 4. Utility code inside Parsons or other libraries - Yes 

65 

66 If you are writing a utility script inside Parsons or another library meant 

67 for broad distribution, you should probably use `DatabaseConnector` type 

68 annotations. This will ensure that your library code will be usable by the 

69 widest possible set of users, not just users on one specific database. 

70 

71 Developer Notes: 

72 This class is an Abstract Base Class (ABC). It's designed to ensure that all classes 

73 inheriting from it implement certain methods, enforcing a consistent interface across 

74 database connectors. 

75 

76 If you need to add a new method to the database connectors, there are three options: 

77 1. Add the method to this ABC and implement it for all databases. 

78 2. Add the method to this ABC and implement it for some databases while adding stubs for 

79 others. 

80 3. Implement the method on a specific database connector without touching the ABC. 

81 

82 If you go the second route, you can add a stub method like this: 

83 

84 .. code-block:: python 

85 

86 def new_method(self, arg1, arg2): 

87 raise NotImplementedError("Method not implemented for this database connector.") 

88 ``` 

89 

90 This communicates clearly to users that the method does not exist for certain connectors. 

91 

92 If you go the third route, remember that you're responsible for making sure your new 

93 method matches the existing methods in other database connectors. For example, if you're 

94 adding a method that already exists in another connector, like Redshift, you need to ensure 

95 your new method behaves the same way and has the same parameters with the same types in the 

96 same order. See the note below for more detail. 

97 

98 Note: 

99 The Python type system (as of 3.10.6) will not stop you from breaking the type contract 

100 of method signatures when implementing a subclass. It is up to the author of a database 

101 connector to ensure that it satisfies this interface. Be careful to, for example, not 

102 change the types of the parameters or leave out optional parameters that are specified 

103 in the interface. 

104 

105 Any such inconsistencies can cause unexpected runtime errors that will not be caught by 

106 the type checker. 

107 

108 It is safe to add additional features to subclasses, such as new methods or extra *optional* 

109 parameters to specified methods. In general adding new methods is safe, but adding optional 

110 parameters to methods specified in the interface should be considered bad practice, because 

111 it could result in unexpected behavior. 

112 

113 Example usage: 

114 

115 .. code-block:: python 

116 

117 def my_function(db: DatabaseConnector, data: Table): 

118 # Your code here, using the db object 

119 

120 # Pass an instance of a class that inherits from DatabaseConnector, e.g. Redshift 

121 my_function(some_db_instance, some_data) 

122

127 """Check if a table or view exists in the database. 

128 

129 `Args:` 

130 table_name: str 

131 The table name and schema (e.g. ``myschema.mytable``). 

132 

133 `Returns:` 

134 boolean 

135 ``True`` if the table exists and ``False`` if it does not. 

136 """ 

137 pass

139 @abstractmethod 

140 def copy(self, tbl: Table, table_name: str, if_exists: str): 

141 """Copy a :ref:`parsons-table` to the database. 

142 

143 `Args`: 

144 tbl (Table): 

145 Table containing the data to save. 

146 table_name (str): 

147 The destination table name (ex. ``my_schema.my_table``). 

148 if_exists (str): 

149 If the table already exists, either ``fail``, ``append``, ``drop`` 

150 or ``truncate`` the table. 

151 """ 

152 pass

154 @abstractmethod 

155 def query(self, sql: str, parameters: Optional[Union[list, dict]] = None) -> Optional[Table]: 

156 """Execute a query against the database. Will return ``None`` if the query returns empty. 

157 

158 To include python variables in your query, it is recommended to pass them as parameters, 

159 following the `psycopg style 

160 <http://initd.org/psycopg/docs/usage.html#passing-parameters-to-sql-queries>`. 

161 Using the ``parameters`` argument ensures that values are escaped properly, and avoids SQL 

162 injection attacks. 

163 

164 **Parameter Examples** 

165 

166 .. code-block:: python 

167 

168 # Note that the name contains a quote, which could break your query if not escaped 

169 # properly. 

170 name = "Beatrice O'Brady" 

171 sql = "SELECT * FROM my_table WHERE name = %s" 

172 db.query(sql, parameters=[name]) 

173 

174 .. code-block:: python 

175 

176 names = ["Allen Smith", "Beatrice O'Brady", "Cathy Thompson"] 

177 placeholders = ', '.join('%s' for item in names) 

178 sql = f"SELECT * FROM my_table WHERE name IN ({placeholders})" 

179 db.query(sql, parameters=names) 

180 

181 `Args:` 

182 sql: str 

183 A valid SQL statement 

184 parameters: Optional[list] 

185 A list of python variables to be converted into SQL values in your query 

186 

187 `Returns:` 

188 Parsons Table 

189 See :ref:`parsons-table` for output options. 

190 """ 

191 pass
